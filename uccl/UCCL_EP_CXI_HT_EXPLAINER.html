Paper Anchor

The main architecture reference for this codebase is arXiv:2512.19849, "UCCL-EP: Portable Expert-Parallel Communication". The paper's central design is the thread that ties this page together: GPU kernels do token-routing work and enqueue compact communication commands, while CPU proxy threads issue the actual network operations and preserve the ordering/completion semantics expected by EP kernels.

Paper idea

Move the hardware-specific network interface work out of GPU kernels and into CPU proxies.

Code consequence

When debugging, travel from kernel command generation to D2H FIFO packing, proxy dequeue/post, transport completion, and FIFO ack.

CXI extension

This branch keeps the paper's proxy architecture but swaps verbs-specific operations for libfabric/CXI writes, atomics, MRs, and CQs.

Mental Model

UCCL EP uses GPU kernels to decide what should move, but CPU proxy threads perform the network operations. The GPU writes compact commands into per-proxy FIFO queues. The CPU proxy drains those commands, posts CXI writes and atomics, polls completions, and only then acknowledges the FIFO slot back to the GPU.

GPU kernel

Computes routing and enqueues compact commands: WRITE, ATOMIC, QUIET, BARRIER.

D2H FIFO

Host-visible queue used as the command handoff between GPU and CPU proxy.

CPU proxy

Converts commands into transport operations and manages ordering/completions.

CXI/libfabric

Executes endpoint-bound RMA writes and native atomics against remote MR offsets.

Transport forced with UCCL_EP_TRANSPORT=cxi Provider forced with FI_PROVIDER=cxi Current Apertus shape uses NUM_MAX_NVL_PEERS=4

Code File Map

The EP path is split across Python setup, C++ runtime/proxy ownership, CUDA kernels, D2H queue packing, and transport implementations. When a failure looks like CXI, first identify which layer last had a valid invariant.

ep/bench/buffer.py

Python-facing Buffer object. Allocates or receives RDMA scratch, wires proxy atomic buffers into the runtime, and exposes dispatch/combine entry points.

ep/src/uccl_ep.cc

Nanobind runtime. Stores rdma_buffer_ptr and atomic_buffer_ptr, launches HT and LL kernels, and exposes buffer/proxy hooks.

ep/src/internode.cu

High-throughput internode dispatch/combine kernels. This is where NUM_MAX_NVL_PEERS, RDMA/NVL rank layout, metadata matrices, and packed routing live.

ep/src/internode_ll.cu

Low-latency internode kernels. Uses the same proxy/FIFO/transport idea with different packet shape, LL metadata, and tighter command packing.

ep/include/ring_buffer.cuh

Defines TransferCmd, command type bits, LL expert/token packing, and write-offset decoding rules.

ep/include/d2h_queue_device.cuh

GPU-side command packer for the D2H queue. Important because the host proxy decodes exactly this 128-bit command shape.

ep/src/proxy.cpp

CPU proxy loop. Dequeues GPU commands, posts verbs or CXI operations, polls completions, and acks FIFO slots only when their visible effect is complete.

ep/src/cxi_transport.cpp

libfabric/CXI transport. Owns provider/domain/endpoint/CQ/AV setup, endpoint-bound MRs, offset-based RMA writes, atomics, and CQ polling.

ep/include/ep_config.hpp

Buffer sizing and layout hints for normal HT and low-latency modes. Audit this when rank count or buffer shape changes.

High-Throughput Rank Layout

High-throughput internode mode groups ranks by local NVL peer index. A rank communicates with the same local GPU index on remote nodes. On 4-GPU Apertus nodes, rank 0 pairs with rank 4, rank 1 with rank 5, and so on.

Local peers
same local index internode peer shape-sensitive local peer count
The original HT path had several places where "local peers" effectively meant eight. That matches the original DeepSeek/DeepEP server shape, but it is not known to be an algorithmic speed requirement. Treat hidden 8-peer coupling as a portability bug until proven otherwise.

Low-Latency Path

LL is not a separate transport stack. It uses different kernels and a different buffer layout, but it still relies on the same GPU-to-proxy command handoff and the same CXI transport backend. That makes LL a useful first validation target before HT tuning.

1. Python setup

ep/bench/buffer.py creates the Buffer and connects proxy atomic memory.

2. Runtime launch

ep/src/uccl_ep.cc calls internode_ll::dispatch or combine.

3. LL kernel

ep/src/internode_ll.cu writes compact TransferCmd entries.

4. Proxy post

ep/src/proxy.cpp decodes LL commands and posts CXI writes/atomics.

5. Counter visible

Remote counters in atomic_buffer_ptr signal that the peer can consume data.

Buffer Lifecycle

The code uses two logically different internode memory regions: the main RDMA payload/scratch buffer and the atomic/signaling buffer. CXI must register both memory regions with endpoint-bound MRs and all remote addressing must be expressed as offsets from the registered base.

Main RDMA Buffer

buffer.pyAllocates or obtains scratch through ep.get_rdma_buffer, then calls runtime set_rdma_buffer.
uccl_ep.ccStores rdma_buffer_ptr, resets it, exposes IPC handles, and passes it to HT/LL kernels.
internode*.cuComputes source and destination offsets relative to the base pointer, not absolute remote pointers.
cxi_transport.cppRegisters HMEM or host memory and sends fi_write to remote MR offsets.

Atomic / Signaling Buffer

uccl_proxy.cppAllocates one shared atomic buffer per proxy group and defensively initializes proxy pointers.
buffer.pyCalls connect_atomic_buffer so the runtime and all proxies share the same signaling memory.
ep_config.hppPlaces LL signaling buffers in atomic_buffer_ptr and guards against overflowing kAtomicBufferSize.
proxy.cppPosts native CXI atomics for counters and barriers, then maps completions back to FIFO slots.
Cleanup/order risk: a cached notify or atomic buffer pointer must not outlive the allocation it points at. When debugging shutdown crashes, inspect proxy ownership, set_atomic_buffer_ptr, and any cached notification state before changing CXI completion logic.

GPU FIFO Command Flow

The GPU pushes commands into FIFO slots. The proxy must not pop a WRITE slot until the operation that makes the command semantically complete has completed. For normal-mode write+counter commands, that means the fenced signaling atomic, not merely the payload write.

CPU Proxy Methodology

The proxy loop is deliberately simple: progress transport completions, notify GPU-visible FIFO state, dequeue new GPU commands, then do normal-mode barrier checks. The hard part is preserving the same completion meaning across verbs and CXI.

StateMeaningBug if wrong
fifo_pending_ FIFO slots accepted from GPU but not yet semantically complete. GPU waits forever if a completed WR never pops the slot.
acked_wrs_ WR ids whose transport-side completion now permits FIFO progress. Early ack corrupts data; missing ack causes hangs.
cxi_outstanding_ops_ Number of CXI operations that still need CQ polling. QUIET may return too early or poll forever.
quiet_wr / barrier_wr Special FIFO command WR ids that must pop only after control completion. QUIET/BARRIER slots remain stuck if CXI bypasses verbs setup.

WR Completion Semantics

A WR is a work request. The proxy attaches a wr_id to each transport operation so the CQ completion can be mapped back to a GPU FIFO slot. In CXI normal mode a single GPU command can become two transport operations: payload write, then fenced signaling atomic.

WRITE only

Use the FIFO command wr_id on the CXI write. Pop when write completes.

WRITE + counter

Use an internal id for the write. Use the FIFO command wr_id for the fenced atomic.

QUIET

Drain outstanding CXI completions, then ack the QUIET command's FIFO slot.

The important invariant is not "a CQ entry exists." The invariant is "the command's visible effect has completed enough for the GPU to proceed."

CXI Normal-Mode Barrier

The verbs path used RDMA write-with-immediate control messages. CXI normal mode now uses a small host MR and native CXI atomics for the internode part while keeping the existing local shared-memory barrier on each node.

8-Local-Rank Assumptions

Many HT failures can look like CXI/provider problems while actually coming from hidden 8-GPU-node assumptions. Always check this class of bug early when running on 4-GPU nodes.

What failed here

The dispatch sender read is_token_in_rank as a raw uint64_t. With four local peers, that reads eight bool bytes from a four-entry group and crosses routing groups.

What fixed it

Pack exactly NUM_MAX_NVL_PEERS bool values into the byte layout consumed by SourceMeta, leaving unused bytes zero.

Other places to audit

Peer matching with MAX_NUM_GPUS, buffer sizing, local-index metadata, dispatch/combine matrices, and vectorized bool loads.

Failure shapes

CUDA misaligned-address, FIFO waits, wrong peer connections, bad route metadata, correctness hash mismatch, or hangs during tuning.

8-node combine winner

The best printed 8-node combine tuning point was SMs 24, NVL chunk 1, RDMA chunk 8, with about 5.07-5.09 GB/s RDMA and 45.99-46.14 ms transmit across the node-local summaries.

Bigger was not better

In the NVL chunk 7 sweep, RDMA chunks 12, 20, and 28 dropped to roughly 2.7-2.9 GB/s, while chunks 8, 16, 24, and 32 stayed near 4.8-5.0 GB/s.

Debugging consequence

Treat chunk size as a discrete scheduling/proxy/provider tuning point. Do not assume increasing the RDMA chunk monotonically improves CXI bandwidth.

Debugging Ladder

The productive order is to prove the transport primitive, then the proxy command lifecycle, then the kernel layout. Avoid jumping directly from a CUDA symptom to a CXI conclusion.

  1. Force CXI explicitly: USE_LIBFABRIC_CXI=1, UCCL_EP_TRANSPORT=cxi, FI_PROVIDER=cxi.
  2. Build for the node shape: NUM_MAX_NVL_PEERS=4 on current 4-GPU Apertus nodes.
  3. Run the smallest low-latency two-node correctness case.
  4. Run small HT with CUDA_LAUNCH_BLOCKING=1 to localize kernel failures.
  5. If a FIFO slot hangs, inspect wr_id, acked_wrs_, fifo_pending_, and special QUIET/BARRIER WR state.
  6. If CUDA reports misalignment, inspect vectorized loads and hidden 8-local-rank assumptions before blaming libfabric.
  7. Only after small HT passes, run larger HT without CUDA_LAUNCH_BLOCKING=1.

Command wire format

Source: ep/include/ring_buffer.cuh

struct TransferCmd {
  CmdType cmd_type;      // WRITE, ATOMIC, QUIET, BARRIER plus mode bits
  uint8_t dst_rank;
  union {
    struct {
      uint32_t atomic_val : 8;
      uint32_t bytes : 24;
    };
    uint32_t bytes_and_val;
  };
  uint32_t req_rptr;     // remote buffer offset or counter offset
  union {
    uint32_t req_lptr;   // local buffer offset for WRITE
    int value;           // value for ATOMIC
  };
  union {
    uint16_t expert_idx;     // LL expert/token packing
    uint16_t atomic_offset;  // normal-mode counter slot
  };
};

4-peer-safe routing bool packing

Source: ep/src/internode.cu

__device__ __forceinline__ uint64_t
pack_is_token_in_nvl_ranks(bool const* values) {
  uint64_t packed = 0;
#pragma unroll
  for (int i = 0; i < NUM_MAX_NVL_PEERS; ++i) {
    packed |= static_cast<uint64_t>(values[i]) << (i * 8);
  }
  return packed;
}
Proxy loop shape

Source: ep/src/proxy.cpp

while (!stop) {
  poll_cxi_completions();     // CQ entries -> acked_wrs_
  notify_gpu();               // completed WR ids pop FIFO slots
  post_gpu_command(tail, seen);
  barrier_check();            // normal-mode barrier progress
}
WRITE plus signaling atomic completion

Source: ep/src/proxy.cpp and ep/src/cxi_transport.cpp

// One GPU FIFO command may become two CXI operations.

post_write(kCxiInternalWrId, local_off, remote_off, bytes);

  rdma_buffer_ptr = reinterpret_cast<void*>(addr);

void set_atomic_buffer_ptr(void* ptr) {
